## 1. Push to GitHub Repository

### 1.1 — Initial Setup (First Time)

```bash
# 1. Initialise git (if not already done)
git init

# 2. Add remote origin (replace with your repo URL)
git remote add origin https://github.com/<your-org>/ai-candidate-discovery.git

# 3. Verify remote
git remote -v
```

### 1.2 — Verify `.gitignore` is Correct

The project `.gitignore` already excludes:

| Excluded Path | Reason |
|---|---|
| `node_modules/` | Dependencies, re-installed by CI |
| `dist/` / `build/` | Build artifacts, generated by CI |
| `.env` | Secrets — **never commit** |
| `uploads/` | User-uploaded files |
| `chroma_data/` | Vector DB data |
| `venv/` / `.venv/` | Python virtualenv |
| `__pycache__/` / `*.pyc` | Python bytecode |

> ⚠️ **Important**: Always run `git status` before committing to confirm no `.env` files are staged.

### 1.3 — Day-to-Day Git Workflow

```bash
# Create a feature branch
git checkout -b feature/your-feature-name

# Stage changes
git add .

# Commit with a conventional message
git commit -m "feat(backend): add resume scoring endpoint"

# Push the branch
git push origin feature/your-feature-name

# Open a Pull Request on GitHub → target: main
```

### 1.4 — Commit Message Conventions

Use **Conventional Commits** for clean, automated changelogs:

```
<type>(<scope>): <short description>

Types:  feat | fix | docs | style | refactor | test | chore | ci
Scopes: frontend | backend | ai-service | docker | ci | docs
```

**Examples:**
```
feat(frontend): add candidate comparison modal
fix(backend): handle null resume text in parser
chore(docker): bump mongo image to 7.0.6
ci: add Docker image build step to CD pipeline
```

### 1.5 — Branch Strategy (Git Flow Lite)

```
main          ← production-ready, protected
  └── staging ← pre-production integration testing
        └── feature/* ← individual feature branches
        └── fix/*     ← bug fix branches
        └── chore/*   ← maintenance/infra branches
```

| Branch | Trigger | Auto-Deploy |
|---|---|---|
| `main` | PR merge | ✅ Production |
| `staging` | PR merge | ✅ Staging |
| `feature/*` | Push | ❌ CI only |

### 1.6 — Branch Protection Rules (GitHub Settings)

Navigate to **Repository → Settings → Branches → Add Rule** for `main`:

- ✅ Require a pull request before merging
- ✅ Require at least **1 approving review**
- ✅ Require status checks to pass before merging
  - `backend` (CI job)
  - `frontend` (CI job)
  - `ai-service` (CI job)
- ✅ Require branches to be up to date before merging
- ✅ Do not allow bypassing the above settings

### 2.3 — Extended CI Workflow (Recommended Upgrade)

Save as `.github/workflows/ci.yml` (replaces the existing one):

```yaml
name: CI

on:
  push:
    branches: [ main, staging ]
  pull_request:
    branches: [ main, staging ]

jobs:
  backend:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Setup Node.js 20
        uses: actions/setup-node@v5
        with:
          node-version: 20
          cache: npm
          cache-dependency-path: backend/package-lock.json
      - name: Install dependencies
        working-directory: backend
        run: npm ci
      - name: Run tests
        working-directory: backend
        run: npm test

  frontend:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Setup Node.js 20
        uses: actions/setup-node@v5
        with:
          node-version: 20
          cache: npm
          cache-dependency-path: frontend/package-lock.json
      - name: Install dependencies
        working-directory: frontend
        run: npm ci
      - name: Type check
        working-directory: frontend
        run: npx tsc --noEmit
      - name: Build
        working-directory: frontend
        run: npm run build

  ai-service:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Setup Python 3.11
        uses: actions/setup-python@v5
        with:
          python-version: '3.11'
          cache: pip
      - name: Install dependencies
        working-directory: ai-service
        run: pip install -r requirements.txt
      - name: Run tests
        working-directory: ai-service
        run: pytest --tb=short || true
```

### 2.4 — CD Workflow (Continuous Deployment)

Save as `.github/workflows/cd.yml`:

```yaml
name: CD — Deploy

on:
  push:
    branches: [ main ]

env:
  REGISTRY: ghcr.io
  IMAGE_PREFIX: ghcr.io/${{ github.repository_owner }}/ai-candidate-discovery

jobs:
  build-and-push:
    name: Build & Push Docker Images
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write

    steps:
      - uses: actions/checkout@v4

      - name: Log in to GHCR
        uses: docker/login-action@v3
        with:
          registry: ${{ env.REGISTRY }}
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Set image tag
        id: meta
        run: echo "TAG=$(git rev-parse --short HEAD)" >> $GITHUB_OUTPUT

      - name: Build & push — backend
        uses: docker/build-push-action@v5
        with:
          context: ./backend
          push: true
          tags: |
            ${{ env.IMAGE_PREFIX }}-backend:${{ steps.meta.outputs.TAG }}
            ${{ env.IMAGE_PREFIX }}-backend:latest

      - name: Build & push — frontend
        uses: docker/build-push-action@v5
        with:
          context: ./frontend
          push: true
          tags: |
            ${{ env.IMAGE_PREFIX }}-frontend:${{ steps.meta.outputs.TAG }}
            ${{ env.IMAGE_PREFIX }}-frontend:latest

      - name: Build & push — ai-service
        uses: docker/build-push-action@v5
        with:
          context: ./ai-service
          push: true
          tags: |
            ${{ env.IMAGE_PREFIX }}-ai-service:${{ steps.meta.outputs.TAG }}
            ${{ env.IMAGE_PREFIX }}-ai-service:latest

  deploy:
    name: Deploy to Production
    runs-on: ubuntu-latest
    needs: build-and-push
    environment: production

    steps:
      - uses: actions/checkout@v4

      - name: Deploy via SSH
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.DEPLOY_HOST }}
          username: ${{ secrets.DEPLOY_USER }}
          key: ${{ secrets.DEPLOY_SSH_KEY }}
          script: |
            cd /opt/ai-candidate-discovery
            git pull origin main
            docker compose pull
            docker compose up -d --remove-orphans
            docker image prune -f

      - name: Health check
        run: |
          sleep 15
          curl --fail ${{ secrets.PRODUCTION_URL }}/api/health \
            || (echo "Health check failed!" && exit 1)
```

## 3. Deployment Strategy

### 3.1 — Architecture Overview

```
                    ┌─────────────────┐
                    │   GitHub GHCR   │ ← Docker images
                    └────────┬────────┘
                             │ docker pull
                    ┌────────▼────────┐
                    │  VPS / Cloud VM │
                    │                 │
                    │  ┌───────────┐  │
                    │  │  Nginx    │  │ ← Reverse proxy + SSL
                    │  │ (port 80/ │  │
                    │  │  443)     │  │
                    │  └─────┬─────┘  │
                    │        │        │
           ┌────────┴──┬─────┴────┬───┴────────┐
           │           │          │             │
     ┌─────▼─────┐ ┌───▼───┐ ┌───▼───┐ ┌──────▼─────┐
     │ Frontend  │ │Backend│ │  AI   │ │  MongoDB   │
     │ :5173     │ │ :4000 │ │Service│ │  :27017    │
     │ (Vite/    │ │(Node) │ │ :8000 │ │(persistent │
     │  static)  │ │       │ │(Python│ │  volume)   │
     └───────────┘ └───────┘ └───────┘ └────────────┘
```

### 3.2 — Environments

| Environment | Branch | URL | Auto-Deploy |
|---|---|---|---|
| **Development** | `feature/*` | `localhost` | ❌ Manual (`docker compose up`) |
| **Staging** | `staging` | `staging.yourdomain.com` | ✅ On merge to `staging` |
| **Production** | `main` | `yourdomain.com` | ✅ On merge to `main` (with approval gate) |

### 3.3 — Rolling Update Strategy

Docker Compose on a single VPS uses a simple **rolling restart**:

```bash
# On the server (automated by CD pipeline)
docker compose pull              # pull new images
docker compose up -d             # recreate containers with zero downtime
docker image prune -f            # clean old layers
```

For **zero-downtime** with multiple replicas (future scale-out), migrate to:
- **Docker Swarm** — `docker service update --image`
- **Kubernetes** — `kubectl rollout` with `RollingUpdate` strategy

### 3.4 — Nginx Reverse Proxy Config (Production)

```nginx
# /etc/nginx/sites-available/ai-candidate-discovery

server {
    listen 80;
    server_name yourdomain.com www.yourdomain.com;
    return 301 https://$host$request_uri;
}

server {
    listen 443 ssl http2;
    server_name yourdomain.com www.yourdomain.com;

    ssl_certificate     /etc/letsencrypt/live/yourdomain.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/yourdomain.com/privkey.pem;

    # Frontend (static files)
    location / {
        root /opt/ai-candidate-discovery/frontend/dist;
        try_files $uri $uri/ /index.html;
    }

    # Backend API
    location /api/ {
        proxy_pass         http://localhost:4000;
        proxy_set_header   Host $host;
        proxy_set_header   X-Real-IP $remote_addr;
        proxy_set_header   X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header   X-Forwarded-Proto $scheme;
    }

    # AI Service
    location /ai/ {
        proxy_pass         http://localhost:8000;
        proxy_set_header   Host $host;
        proxy_read_timeout 120s;
    }
}
```

## 5. Rollback Procedure

### 5.1 — Immediate Rollback (Docker Image Tag)

Each deployment tags images with the Git short SHA:

```bash
# On the production server
PREVIOUS_TAG=a1b2c3d

docker pull ghcr.io/<org>/ai-candidate-discovery-backend:$PREVIOUS_TAG
docker pull ghcr.io/<org>/ai-candidate-discovery-frontend:$PREVIOUS_TAG
docker pull ghcr.io/<org>/ai-candidate-discovery-ai-service:$PREVIOUS_TAG

IMAGE_TAG=$PREVIOUS_TAG docker compose up -d
```

### 5.2 — Git Revert (Safe Rollback)

```bash
# Never force-push to main — use revert
git revert HEAD --no-edit
git push origin main
# This triggers the CD pipeline → automatically redeploys the reverted code
```

### 5.3 — Database Backup & Restore

```bash
# Backup before any risky migration
docker exec mongodb mongodump --out /backup/$(date +%Y%m%d_%H%M%S)

# Restore if needed
docker exec mongodb mongorestore /backup/<timestamp>
```

## 6. Monitoring & Health Checks

### 6.1 — Add a Health Check Endpoint (Backend)

Add to `backend/src/index.js`:

```javascript
app.get('/api/health', (req, res) => {
  res.json({
    status: 'ok',
    timestamp: new Date().toISOString(),
    version: process.env.npm_package_version,
    services: {
      database: mongoose.connection.readyState === 1 ? 'connected' : 'disconnected',
    }
  });
});
```

### 6.2 — Docker Compose Health Checks

Add to `docker-compose.yml`:

```yaml
backend:
  healthcheck:
    test: ["CMD", "curl", "-f", "http://localhost:4000/api/health"]
    interval: 30s
    timeout: 10s
    retries: 3
    start_period: 40s

  healthcheck:
    test: ["CMD", "curl", "-f", "http://localhost:8000/health"]
    interval: 30s
    timeout: 15s
    retries: 3
    start_period: 60s
```
